Introduction

It’s time to get productive with Ansible Playbooks for Confluent Platform. In this module, you will configure a set of hosts in a development environment to run Confluent services. After this module, you will be able to quickly provision a development environment that empowers your engineers to experiment with real-time, event-driven applications.

Activity

The purpose of this activity is to deploy Confluent to a set of servers to allow developers to easily experiment and iterate on their event-driven applications. Even though we are only working with a development environment, it is still important to set up basic security with TLS encryption for data in motion, and SASL/PLAIN authentication. Authorization, however, may be unnecessary in this environment.

Ansible pairs nicely with infrastructure tools like Terraform or CloudFormation for disposable development environments. For example: create servers with Terraform, provision Confluent services with Ansible, and then tear down the whole environment with Terraform. In this lab activity, Docker Compose simulates this workflow.

Prerequisites

    source ~/venv/bin/activate
    The prompt should now show (venv) to indicate the virtual environment is active.

  3. Open the file ~/course/docker-compose-dev.yml. This file defines the servers that will act as our target hosts. What do you notice? What do you wonder? Write down a few thoughts an compare your response to the sample response.

    sample response

    First, note that all hosts run from the same "raw" CentOS image. The goal is to run Confluent Playbooks for Ansible to deploy different Confluent components to different identical servers.

    There is one ZooKeeper host, three Kafka broker hosts, one Schema Registry host, one Kafka Connect host, one Control Center host with port 9021 exposed to the local workstation.

Create Certificates

While this is only a development environment, it is still appropriate to encrypt data in motion with TLS/SSL. The Ansible Playbooks for Confluent Platform include some helpful scripts for generating certificates, which have been collected into an Ansible playbook at ~/course/certs/cert-playbook.yml. Note that client applications will have to trust the Certificate Authority that we generate.

  1. Run the certificate playbook with vault password confluent.

    ansible-playbook --ask-vault-pass \
    certs/cert-playbook.yml \
    -i inventories/development/development.yml
    Generated SSL files should now be available under ~/course/certs/generated_ssl_files

Run Confluent Playbooks

Usually you would need to git clone the CP Ansible GitHub repository and checkout the branch corresponding to the version of Confluent Platform you want to use (e.g. 7.2.1-post), or run ansible-galaxy collection install confluent.platform. That has already been done for you. The collection (or repo) is located at ~/.ansible/collections/ansible_collections/confluent/platform.

  1. Change directory to the CP Ansible repo.

    cd ~/.ansible/collections/ansible_collections/confluent/platform

  2. Run the playbook called all.yml against the development host inventory. Remember the vault password is confluent.

    ansible-playbook --ask-vault-pass \
    ./playbooks/all.yml \
    -i ~/course/inventories/development/development.yml
    Pay attention to the Ansible output as the playbook runs to get a sense of what is happening. This will take about 20 minutes.

Validate the Deployment

  1. Check the server.properties file on one of the brokers directly.

    docker-compose \
    -f ~/course/docker-compose-dev.yml \
    exec kafka-0 \
        cat /etc/kafka/server.properties

  2. Open Confluent Control Center in the browser at https://localhost:9021. Then select the cluster controlcenter.cluster.

    Since we are using self signed certificates, you will have to ignore the warning by selecting Advanced → Proceed to localhost (unsafe).

  3. Go to Cluster settings → Cluster defaults and confirm that the correct properties have been configured.

  4. Create a topic called test with default properties by selecting Topics → Add topic with the name test and select Create with defaults.

  5. Launch an instance of the Datagen source connector by selecting Connect → connect-cluster → Add connector → Datagen connector with these general settings:

    • kafka.topic should be set to test to produce to the test topic.

    • quickstart should be set to pageviews to produce fake pageview data.

      1. After configuring the connector, launch by selecting continue → Launch

        This validates that Kafka Connect is functioning.

  6. View the schema for the topic test by navigating to Topics → test → Schema

    This validates that Schema Registry is functioning.

  7. Run a kafkacat consumer client to consume records coming into the test topic from the Datagen connector. Press Ctrl+C to quit when you are ready to move on.

    docker run --rm --tty \
    --network confluent-dev \
    --entrypoint /bin/sh \
    -v /home/training/course/certs/generated_ssl_files/ca.crt:/usr/local/share/ca-certificates/ca.crt \
    edenhill/kafkacat:1.6.0 \
        -c "update-ca-certificates && \
        kafkacat -b kafka-0.dev.confluent:9092 -t test \
            -X security.protocol=sasl_ssl -X sasl.mechanism=PLAIN \
            -X sasl.username=client -X sasl.password=client-secret \
            -C \
            -s key=avro -s value=avro -r https://schema-registry.dev.confluent:8081 \
            -f '\nKey (%K bytes): %k\t\nValue (%S bytes): %s\n\Partition: %p\tOffset: %o\n--\n' "
    1 Docker settings:

    • --rm removes the container after it runs.

    • --tty enables teletypewriter, which means you will be able to see the output from kafkacat in your terminal.

    • -v bind mounts the CA certificate into the kafkacat container so that the kafkacat client will trust the broker and Schema Registry certificates during SSL/TLS encryption.

    • --network specifies the name of the network to run the container in. This allows the kafkacat client to run within the same Docker network as the servers so that they can be reached by hostname.

    • --entrypoint overrides the container’s launch command. In this case, we override the kafkacat command with /bin/sh, which will allow us to update the container’s trusted CA later. The /bin/sh shell runs commands with the -c option, which is why there is a -c before the commands that run in the container. Also, /bin/sh -c takes string as input to execute commands, which is why those commands are in double quotes.

    • edenhill/kafkacat:1.6.0 is the name of the kafkacat Docker image.
    2 update-ca-certificates updates trusted CA certificates with the self-signed CA certificate we generated.
    3 Kafka client settings:

    • -X specifies properties we are passing to the Kafka client, which in this case are related to SASL authentication. We don’t need to specify ssl.ca.location since we updated the container’s trusted CA certs.
    4 Kafkacat options:

    • -C runs kafkacat in Consumer mode.

    • -s defines the serializer/deserializer ("SerDes"). In this case we use a string deserializer for the record key and an avro deserializer for the record value.

    • -r defines the Schema Registry REST API server where avro schemas will be stored and retrieved.

    • -f defines string formatting for the output. See kafkacat -h for more information.

    • -t sets the topic, which in this case is test.

      This validates that clients can reach the brokers and Schema Registry directly.

Activity Debrief

Discussion Questions

  1. What is the importance of Docker Compose in this activity?

    sample response

    Docker Compose is used to provision a set of "raw" CentOS servers against which to run the Confluent Playbooks for Ansible. This simulates a set of networked servers with DNS hostnames running somewhere in the cloud, like a VPC. Because they are all in the same network called confluent-dev, it is possible to run the kafkacat container with docker run --network confluent-dev to simulate a Kafka client running inside the same VPC.

  2. What is the importance of vault.yml in this activity?

    sample response

    The vault.yml file defines variables with sensitive credentials that shouldn’t be exposed in source control. These variables are prefixed with vault_ as a naming convention, and are referenced by other variables.

  3. What is the importance of the group_vars directory in this activity?

    sample response

    The group_vars directory is where we set important variables that the cp-ansible playbooks will use when configuring the host machines.

Summary

In this activity, you provisioned and configured a full Confluent deployment suitable for developers to run their applications and experiments. This deployment included SSL/TLS for transport encryption as well as SASL/PLAIN for basic username-password authentication.

Because this was a developer environment, there were no strict authorization rules put into place. In a later module, you will deploy Confluent with production-grade security, including authorization with Confluent Role Based Access Control.
